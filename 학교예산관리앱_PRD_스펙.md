# 학교 예산 사용 관리 웹앱 — PRD & 기능 명세서

## 0. 문서 정보
- 작성 목적: 개발자 또는 AI 코딩 도구가 바로 실행할 수 있는 요구사항 정의서
- 대상 사용자: 학교 내 여러 교사 + 행정 담당자
- 핵심 한 줄 요약: 프로젝트/사업별로 연도 예산을 등록하고, 누가 언제 얼마를 썼는지, 잔액이 얼마인지 한눈에 관리하는 앱

---

## 1. 요구사항 정의

### 1.1 만들려는 것
학교 예산을 **연도 + 프로젝트(사업)별**로 등록하고, 사용 내역(항목/금액/일자/증빙)을 기록하여 **실시간 잔액**을 확인할 수 있는 웹앱입니다.

### 1.2 사용 목적 체크리스트 (심의 대응용)
| 항목 | 해당 여부 |
|---|---|
| 교사·행정 담당자 단독 행정·업무용 | ✅ 해당 |
| 정규 수업에서 학생과 함께 사용 | ❌ 해당 없음 |
| 학생 개인정보 수집·처리 | ❌ 해당 없음 (예산 데이터만 처리) |
| 가정/과제용 사용 | ❌ 해당 없음 |

> 본 앱은 **교직원 업무용 행정 시스템**으로 분류되며, 학생 개인정보를 다루지 않으므로 학습지원 소프트웨어 심의 대상 여부를 학교 담당자가 별도 확인할 것을 권장합니다.

### 1.3 핵심 기능 목록
1. 회원 관리: 이메일(학교 아이디) + 비밀번호 로그인, 비밀번호 재설정
2. 사용자 역할: 관리자 / 일반 교사 (2단계)
3. 프로젝트(사업)별 연도 예산 배정액 등록
4. 예산 사용 내역 등록 (항목/금액/일자/증빙파일)
5. 실시간 잔액 현황판 (사용액/잔액 자동 계산, 승인 절차 없이 즉시 반영)
6. 증빙 파일(영수증) 첨부 — 이미지/PDF 업로드
7. 엑셀(xlsx) 데이터 내보내기
8. 권한별 접근 제어 (담당 항목 vs 전체)
9. 개인정보처리방침 / 이용약관 페이지 및 푸터

### 1.4 사용자 역할 정의
| 역할 | 권한 범위 |
|---|---|
| **관리자** | 전체 프로젝트/예산 조회·등록·수정·삭제, 전체 사용 내역 수정/삭제, 엑셀 내보내기, 사용자 계정 관리 |
| **일반 교사** | 본인이 담당으로 지정된 프로젝트(항목)만 조회, 해당 항목에 사용 내역 등록/수정(본인 작성분), 본인 담당 범위 엑셀 내보내기 |

> 일반 교사는 자신이 등록한 내역이라도 **삭제는 관리자만 가능**하도록 하여 기록 보존성을 높입니다(필요 시 교사 본인 수정은 허용, 삭제는 관리자 승인 없이 즉시 가능하지만 이력은 보존 — 아래 데이터 모델의 `deleted_at` 참고).

## 2. 화면 흐름

### 2.1 공통
```
[로그인 화면] → [비밀번호 재설정] (필요 시)
      ↓
[메인 대시보드] — 역할에 따라 보이는 범위 다름
```

### 2.2 관리자 화면 흐름
```
대시보드 (전체 프로젝트별 잔액 카드)
  ├─ 프로젝트 관리
  │    ├─ 새 프로젝트(사업) 등록 (연도, 사업명, 배정액, 담당 교사 지정)
  │    ├─ 프로젝트 수정/삭제
  │    └─ 담당자 지정/변경
  ├─ 전체 사용 내역 조회
  │    ├─ 내역 상세 보기 (증빙 파일 포함)
  │    ├─ 내역 수정/삭제 (모든 항목)
  │    └─ 엑셀 내보내기 (필터: 연도/프로젝트/기간)
  ├─ 사용자 관리 (계정 생성/역할 변경/비활성화)
  └─ 개인정보처리방침/이용약관 (푸터)
```

### 2.3 일반 교사 화면 흐름
```
대시보드 (본인 담당 프로젝트별 잔액 카드만 표시)
  ├─ 담당 프로젝트 상세 (배정액 / 사용액 / 잔액)
  ├─ 사용 내역 등록
  │    ├─ 항목명, 금액, 사용일자 입력
  │    └─ 증빙 파일 첨부 (이미지/PDF)
  ├─ 내가 등록한 내역 조회/수정 (삭제는 관리자에게 요청)
  ├─ 담당 범위 엑셀 내보내기
  └─ 개인정보처리방침/이용약관 (푸터)
```

### 2.4 화면 우선순위
> 사용자 확인: "프로젝트 배정액 등록 화면이 가장 먼저 필요합니다."
1순위: 프로젝트/연도 예산 배정액 등록 화면
2순위: 사용 내역 등록 화면 (증빙 첨부 포함)
3순위: 잔액 현황 대시보드
4순위: 엑셀 내보내기
5순위: 사용자/권한 관리

## 3. 기능별 모듈 설계

### 3.1 인증 모듈
- 역할: 로그인/로그아웃, 비밀번호 재설정
- 주요 기능: 이메일+비밀번호 로그인, 이메일 인증 기반 비밀번호 재설정 링크 발송
- 입력 데이터: 이메일, 비밀번호
- 출력 데이터: 세션 토큰, 역할(role) 정보
- 예외 상황: 비밀번호 5회 오류 시 일시 잠금 권장, 미인증 이메일 처리
- 수정 시 주의점: 비밀번호는 반드시 해시 저장(bcrypt 등), 평문 저장 금지

### 3.2 프로젝트(예산) 관리 모듈
- 역할: 연도별 프로젝트(사업) 단위 예산 배정액 등록/관리
- 주요 기능: 프로젝트 생성/수정/삭제, 담당 교사 지정(1명 이상 가능), 연도 필터
- 입력 데이터: 연도, 사업명, 배정액, 담당자 ID 목록, 비고
- 출력 데이터: 프로젝트 목록, 잔액(배정액 − 사용액 합계)
- 예외 상황: 배정액 0 또는 음수 입력 방지, 동일 연도+사업명 중복 경고
- 수정 시 주의점: 배정액 수정 시 기존 사용 내역과의 정합성(과사용 여부) 경고 표시

### 3.3 사용 내역(지출 기록) 모듈
- 역할: 실제 예산 사용 건 등록/조회/수정/삭제
- 주요 기능: 항목명, 금액, 사용일자 입력, 증빙파일 업로드, 즉시 반영(승인 절차 없음)
- 입력 데이터: 프로젝트 ID, 항목명, 금액, 사용일자, 증빙파일, 작성자 ID
- 출력 데이터: 내역 리스트, 합계 금액
- 예외 상황: 잔액 초과 사용 시 경고 문구 표시(차단은 하지 않음, 기록 목적), 파일 형식/용량 제한(예: 10MB 이하, jpg/png/pdf)
- 수정 시 주의점: 교사 본인 작성 건만 본인 수정 가능, 삭제는 관리자만 — 삭제 시 하드 삭제 대신 `deleted_at` 처리(이력 보존) 권장

### 3.4 대시보드(현황판) 모듈
- 역할: 프로젝트별 배정액/사용액/잔액 실시간 계산 및 시각화
- 주요 기능: 카드형 요약, 프로젝트 상세 클릭 시 내역 리스트 이동
- 입력 데이터: 없음(집계 쿼리)
- 출력 데이터: 배정액, 누적 사용액, 잔액, 사용률(%)
- 예외 상황: 데이터 없음(0건) 상태 UI 처리
- 수정 시 주의점: 집계는 항상 `deleted_at IS NULL` 조건으로 계산

### 3.5 엑셀 내보내기 모듈
- 역할: 조회 중인 데이터를 xlsx로 다운로드
- 주요 기능: 필터(연도/프로젝트/기간) 적용된 사용 내역 + 요약 시트 내보내기
- 입력 데이터: 필터 조건
- 출력 데이터: .xlsx 파일 (항목명, 금액, 일자, 작성자, 프로젝트명 컬럼 포함)
- 예외 상황: 데이터 0건일 때 빈 템플릿 다운로드 처리
- 수정 시 주의점: 일반 교사는 본인 담당 프로젝트 범위로만 내보내기 가능하도록 서버단 필터링 필수(클라이언트 필터링만 하면 권한 우회 위험)

### 3.6 권한/보안 모듈
- 역할: 역할 기반 접근 제어(RBAC)
- 주요 기능: 관리자/일반 교사 권한 분기, API 단 권한 검증
- 입력 데이터: 요청자 세션의 role, 담당 프로젝트 매핑
- 출력 데이터: 허용/차단 결과
- 예외 상황: 권한 없는 프로젝트 접근 시도 시 403 처리
- 수정 시 주의점: 프론트엔드 메뉴 숨김만으로 권한 처리하지 말 것 — 반드시 백엔드(API/DB 규칙)에서도 검증

### 3.7 개인정보보호/운영정보 모듈
- 역할: 개인정보처리방침, 이용약관, 책임자 정보 표시
- 주요 기능: 푸터 링크, 팝업 또는 별도 페이지 표시
- 입력 데이터: 정책 텍스트(관리자 입력 또는 고정 템플릿)
- 출력 데이터: 정책 페이지/팝업
- 예외 상황: 정책 미작성 시 임시 안내 문구 노출
- 수정 시 주의점: 기관명, 책임자, 시행일은 학교 담당자가 직접 채워야 함(아래 4장 참고)

## 4. 데이터 모델 (예시 필드명)

> 학생 개인정보는 전혀 포함하지 않으며, 교직원 계정 정보 + 예산 데이터만 다룹니다.

### 4.1 users (사용자 — 교직원)
| 필드 | 타입 | 설명 | 수집 목적 | 접근 권한 |
|---|---|---|---|---|
| id | string | 사용자 고유 ID | 식별 | 시스템 |
| email | string | 로그인 이메일(학교 계정) | 인증 | 본인, 관리자 |
| password_hash | string | 비밀번호 해시 | 인증 | 시스템 전용 |
| name | string | 실명 | 담당자 식별, 책임 소재 | 본인, 관리자 |
| position | string | 직책/부서 | 권한 부여 기준 | 본인, 관리자 |
| role | enum(admin, teacher) | 권한 등급 | 접근 제어 | 관리자 |
| created_at | datetime | 가입일 | 운영 | 관리자 |

- 보관 범위: 재직 기간 + 퇴직 후 행정 기록 보존 기간(학교 내부 규정에 따름, 미정 시 1년 권장)
- 삭제/내보내기 기준: 관리자가 계정 비활성화 가능, 실제 삭제는 회계연도 종료 후 검토

### 4.2 projects (프로젝트/사업)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | 프로젝트 고유 ID |
| year | int | 회계연도 |
| name | string | 사업명 |
| allocated_amount | decimal | 배정액 |
| owner_ids | array(user_id) | 담당 교사 목록 |
| note | string | 비고 |
| created_at | datetime | 등록일 |

### 4.3 expenses (사용 내역)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | 내역 고유 ID |
| project_id | string | 소속 프로젝트 |
| item_name | string | 사용 항목명 |
| amount | decimal | 사용 금액 |
| used_date | date | 사용일자 |
| receipt_file_url | string | 증빙 파일 경로 |
| created_by | user_id | 작성자 |
| created_at | datetime | 등록 시각 |
| deleted_at | datetime (nullable) | 삭제 처리 시각(이력 보존용) |

- 수집 목적: 예산 집행 투명성, 잔액 산출
- 보관 범위: 최소 회계연도 종료 후 3~5년(학교 회계 규정 확인 필요 — **교사 확인 필요 지점**)
- 접근 권한: 작성자 본인(수정), 관리자(전체 수정/삭제)
- 삭제 기준: 일반 교사는 삭제 불가, 관리자만 가능(소프트 삭제 권장)

## 6. 개인정보보호 구현 계획
1. **수집 최소화**: 학생 관련 정보 일체 미수집. 교직원 이름·직책·이메일만 수집(권한 부여 목적).
2. **권한 분리**: 역할 기반 접근 제어로 본인 담당 범위 외 데이터 비노출.
3. **파기 절차**: 계정 퇴직/전출 시 관리자가 비활성화 → 회계연도 종료 후 일정 기간 뒤 삭제(학교 내부 규정 확인 후 확정).
4. **안전성 확보조치**: 비밀번호 해시 저장, 전송 구간 HTTPS 적용, 증빙 파일 업로드 시 파일 검증(확장자/용량 제한).
5. **정책 고지**: 모든 화면 푸터에 개인정보처리방침/이용약관/책임자 정보/저작권 문구를 표시합니다.
   - 푸터 표기: `개인정보처리방침 | 이용약관 | 책임자: 조정연 / 서울언북초등학교 / 02-514-5981 | © 2026 ingyu's AI world. All rights reserved.`
   - 팝업 구성: 개인정보처리방침(8개 조항)과 이용약관은 각각 별도 팝업으로 분리 표시

## 부록 A. 데이터베이스 스키마 (SQL)

> 별첨 파일: `schema.sql` (Antigravity 작업 시 함께 첨부하면 됩니다)

PostgreSQL(Supabase) 기준으로 작성되었으며 다음을 포함합니다.
- `users`, `projects`, `project_owners`, `expenses` 4개 테이블
- 잔액 자동 계산용 `project_balance` 뷰
- 역할 기반 접근 제어를 위한 Row Level Security(RLS) 정책
  - 관리자: 전체 프로젝트/사용 내역 조회·수정·삭제 가능
  - 일반 교사: 담당 프로젝트만 조회, 본인 작성 내역만 등록/수정 가능 (삭제 불가)
- 증빙 파일은 DB에 직접 저장하지 않고 Storage 업로드 후 URL만 `expenses.receipt_file_url`에 저장
- Firebase 사용 시 변환 가이드(컬렉션 구조)는 파일 하단 주석 참고

전체 내용은 `schema.sql` 파일을 참고하세요. (본 PRD 3장 데이터 모델과 1:1로 대응됩니다.)
